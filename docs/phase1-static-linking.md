# Phase 1 補足: 静的リンクと CGO_ENABLED=0 について

学習計画 `podman-aws-learning-plan.md` の Phase 1 で出てくる「静的リンク必須」の解説。Containerfile に `ENV CGO_ENABLED=0 GOOS=linux` を書く理由をきちんと理解するための資料。

---

## 静的リンク・動的リンクとは

プログラムが動くには、自分のコードだけでなく **外部ライブラリ** が必要なことが多い(C言語の `printf` も標準 C ライブラリ = libc に入っている)。このライブラリの組み込み方に2種類ある。

### 動的リンク(dynamic linking)
- プログラムには「**libc を使うよ**」というメモだけが入っている
- 実行時にOSが `/lib/libc.so` のような実体を探して読み込む
- ファイルサイズが小さい / 複数のプログラムで libc を共有できる
- **ただし: 実行環境に libc が無いとプログラムは起動できない**

```
あなたのバイナリ ──(実行時に探す)──> OSの /lib/libc.so
       ↑
       「libc が必要」とメモだけ持っている
```

### 静的リンク(static linking)
- 必要なライブラリを **全部バイナリの中にコピーして埋め込む**
- ファイルサイズは大きくなる
- **でも: 実行環境に何も無くてもそのバイナリ単体で動く**

```
あなたのバイナリ [自分のコード + libcのコピー全部]
       ↑
       これ1個で完結。OSにライブラリを要求しない
```

## なぜ distroless で問題になるか

`gcr.io/distroless/static-debian12` の `static` という名前がヒント。このイメージは「**静的リンクされたバイナリしか動かさない前提**」で作られていて、**libc すら入っていない**(極限まで小さく安全にするため)。

ここに動的リンクされたバイナリを置いて起動すると、OS が libc を探しに行くも見つからず、

```
exec /app: no such file or directory
```

のような **謎エラーで起動と同時にクラッシュ**する(ファイルはあるのに「no such file」と出るのが初心者殺し)。

---

## Go と CGO の関係

Go 自体は通常、標準ライブラリだけで完結する静的リンクが基本。**ところが** Go には CGO という「Go から C のコードを呼ぶ機能」があり、これが **デフォルトで有効** になっている。

CGO が有効だと、Go の標準ライブラリの一部(`net` パッケージの名前解決まわりなど)が **こっそり libc を動的リンクで呼ぶ** ようになり、結果として「distroless/static で動かない動的リンクバイナリ」が出来上がる。

これを防ぐ呪文が:

```
ENV CGO_ENABLED=0 GOOS=linux
```

- `CGO_ENABLED=0` → CGO を完全に切る = 100% 純粋な Go コードだけでビルド = 完全静的リンク
- `GOOS=linux` → Linux 向けにビルド(macOS から作る時のクロスコンパイル指定)

---

## ついでに `-ldflags="-s -w"` とは

```
RUN go build -ldflags="-s -w" -o /out/app .
```

これはバイナリサイズを小さくするオプション。

- `-s` → シンボルテーブルを削除
- `-w` → DWARF デバッグ情報を削除

両方付けると **バイナリが 20〜30% くらい小さくなる**。本番運用で gdb デバッグしない前提なら付けるのが定番。学習目的なら無くても動作には影響しない。

---

## まとめ

| 項目 | 役割 |
|------|------|
| `CGO_ENABLED=0` | C ライブラリを使わせない → 完全静的リンク化 |
| `GOOS=linux` | Linux 向けバイナリを作る |
| `-ldflags="-s -w"` | バイナリを小さくする(任意) |
| distroless/static | 静的バイナリ専用の極小ベースイメージ |

「**CGO を切る = libc 依存を断つ = distroless/static で動く**」というのが一連の流れ。CGO 切り忘れが Go × distroless の鉄板ハマりどころなので、Phase 1 で一度自分の手で踏んでみると印象に残る(切らずにビルドして動かないことを確認 → 切ってリビルドして動くことを確認、の順でやると効果的)。
